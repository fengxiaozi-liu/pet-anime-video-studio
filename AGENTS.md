# AGENTS.md

面向本仓库中 AI 编码代理和协作者的工作指南。除非用户明确要求，所有改动都应遵循“谨慎、简单、可验证、少改动”的原则。

## 项目背景

Video Studio 是一个基础视频创作工具，当前形态是 FastAPI 后端加服务端模板页面：

- 后端在 `backend/app/`，入口为 `backend/app/main.py`
- 前端模板在 `front/templates/`
- 前端静态资源在 `front/static/`
- 默认配置在 `config.yaml`
- 测试在 `backend/tests/`
- 页面由 FastAPI 直接渲染 Jinja2 模板，不是独立 SPA 工程

项目目标是通用视频工具，不再绑定宠物、萌宠等单一内容品类。新增文案、默认 prompt、导出标题、标签和 README 说明都应保持通用视频创作定位。

## 1. 写代码前先想清楚

不要假设，不要掩盖困惑，要主动说明取舍。

实现前应做到：

- 明确说明自己的假设；如果不确定，就问清楚。
- 如果需求有多种理解，要列出来，不要悄悄选择其中一种。
- 如果存在更简单的方案，要说出来；必要时对过度设计提出异议。
- 如果有地方不清楚，停下来，指出困惑点并询问。

对于本项目，尤其要确认：

- 改动影响的是后端 API、服务端页面、素材配置、任务流程，还是导出包。
- 文案是否面向通用视频创作，而不是某个具体垂类。
- 配置变更是否会影响本地开发、Docker 启动或已有数据库。

## 2. 简单优先

只写解决问题所需的最少代码，不做推测性扩展。

- 不添加用户没有要求的功能。
- 不为单次使用代码增加抽象。
- 不添加未被要求的“灵活性”或“可配置性”。
- 不为不可能发生的场景写复杂错误处理。
- 如果 200 行代码可以写成 50 行，应重新简化。

自检问题：

> 资深工程师会不会觉得这段实现过度复杂？

如果答案是“会”，就先简化。

## 3. 外科手术式改动

只触碰必须触碰的文件，只清理自己造成的问题。

编辑现有代码时：

- 不顺手“优化”无关代码、注释或格式。
- 不重构没有坏掉的代码。
- 匹配现有代码风格，即使你个人会采用别的写法。
- 如果发现无关的死代码或乱码，先提出来，不要擅自删除。

当你的改动造成孤儿代码时：

- 删除由你的改动造成的未使用 import、变量、函数。
- 不删除改动前就存在的死代码，除非用户明确要求。

检验标准：

> 每一行变更都应该能直接追溯到用户请求。

## 4. 目标驱动执行

把任务转化为可验证目标，并循环到验证通过。

示例：

- “添加校验” → 为非法输入写测试，再让测试通过。
- “修复 bug” → 写出能复现 bug 的测试，再修复并验证。
- “重构 X” → 重构前后都确认相关测试通过。

多步骤任务应先给出简短计划：

```text
1. [步骤] → 验证：[检查方式]
2. [步骤] → 验证：[检查方式]
3. [步骤] → 验证：[检查方式]
```

好的成功标准可以让代理独立推进。模糊标准，例如“让它能用”，通常需要进一步澄清。

## 项目开发约定

### 后端

- 使用 FastAPI、Pydantic、Jinja2Templates。
- API 入口集中在 `backend/app/main.py`。
- 配置读取逻辑在 `backend/app/config.py`，默认值和 `config.yaml` 要保持一致。
- 任务、素材、provider、assistant 相关逻辑应优先使用现有 service/repository 模式。
- 不要绕开现有 `JobApplicationService`、`MaterialAssetService`、provider config service 等应用层封装，除非需求明确要求。

### 前端

- 当前没有前端构建链路，直接编辑 `front/templates/*.html` 和 `front/static/*.css|js`。
- 页面应保持工具型工作台风格：信息清晰、操作直接、少营销话术。
- 首页应作为通用视频创作入口，不要再使用宠物项目叙事。
- 修改页面文案时，优先使用清晰中文，避免引入乱码或混合编码内容。
- 不要为一个简单页面改动引入新框架或构建工具。

### 配置与运行数据

- `.venv/`、`.pytest_cache/`、`front/htmlcov/`、`uploads/`、`outputs/`、`.data/` 属于本地或运行期产物，不应提交。
- `config.yaml` 是默认配置文件；涉及默认数据库名、路径、provider 默认配置时，同步检查 `backend/app/config.py`。
- 不要删除用户已有运行数据，除非用户明确要求。

### 测试与验证

优先运行最小相关测试：

```powershell
cd backend
..\.venv\Scripts\python.exe -m pytest tests\test_export_package.py -q
```

更大范围验证：

```powershell
cd backend
..\.venv\Scripts\python.exe -m pytest tests -q
```

后端启动验证：

```powershell
cd backend
..\.venv\Scripts\python.exe -m uvicorn app.main:app --host 127.0.0.1 --port 8000
```

健康检查期望：

```json
{"status":"healthy","service":"video-studio"}
```

如果完整测试因仓库中已有无关改动失败，要在最终说明中明确区分：

- 本次改动已验证通过的部分
- 失败的测试或命令
- 失败是否与本次改动相关

## 最终回复要求

完成任务后，简要说明：

- 改了哪些文件
- 为什么这样改
- 运行了哪些验证
- 是否还有残留风险或未验证项

不要把无关实现细节倾倒给用户。结论要清楚，信息要足够复现。
