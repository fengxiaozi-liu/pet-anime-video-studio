# 任务系统与 ffmpeg 链路讨论文档

## 说明

本轮只更新规划文档，不修改前端或后端代码。

本文档用于统一后续讨论基线，聚焦三个方向：

- 任务系统从 `jobs.json + 应用内后台线程` 迁移到 `MySQL + 独立 Worker`
- 前端任务流 UI 的后续改造计划，范围仅覆盖任务相关页面与交互
- 本地 `ffmpeg` 链路的后续方向：短期保留，但从“本地图片轮播渲染器”收缩为“后处理与兜底模块”

后续实现应以本文档为讨论基线。

## 背景与现状

### 当前任务创建方式

- 现在任务只有点击“生成视频”才会创建
- 这一点保留，不提供“手动新增空任务”的入口

### 当前任务存储与异步模型

- 当前任务存储在 `.data/jobs.json`
- 当前任务元数据由 `JobStore` 读写本地 JSON 文件
- 当前异步执行不是正式队列，而是应用内 `threading.Thread`
- 当前任务提交后会直接在应用进程里起后台线程执行
- 当前没有暂停任务接口
- 当前没有删除任务接口

### 当前渲染链路事实

当前项目实际存在两条渲染链路：

#### 1. local 链路

- `local` 链路当前不是调用云端 provider
- 它是本地渲染链路
- 现有实现会基于图片按分镜生成视频片段
- 然后使用 `ffmpeg` 做以下工作：
  - 单镜头片段生成
  - 镜头转场
  - 字幕烧录
  - BGM 混音
  - 最终成片输出

结论：

- 当前本地 `ffmpeg` 链路承担的是完整本地渲染职责，而不只是后处理

#### 2. cloud 链路

- `cloud` 链路理论上由 provider 直接产出最终视频文件
- 当前 provider 接口的抽象是 `provider.render(ctx)`
- 从现有代码状态看，大部分 provider 仍是 stub，尚未完整落地

结论：

- 当前系统设计上允许云端 provider 直接生成最终成片
- 但从工程成熟度看，本地链路仍是当前更完整的实现

## 目标任务系统

### 总体目标

- 任务必须迁移到 MySQL 持久化
- 执行模型改为独立 Worker，而不是继续沿用应用内线程
- 任务完成、失败、暂停、删除都应有明确状态和持久化记录

### 任务创建原则

- 任务不能手动“新增”
- 任务只能通过点击“生成视频”创建
- `POST /api/jobs` 仍是唯一任务创建入口
- 该接口后续语义应改为“入库并排队”

### 目标状态流转

后续任务状态统一规划为：

- `queued`
- `running`
- `paused`
- `done`
- `error`
- `deleted`

### 目标执行模型

- FastAPI 负责接收生成请求、入库、查询、控制任务
- 独立 Worker 负责轮询 MySQL 中的待执行任务
- Worker 抢到任务后更新状态为 `running`
- 执行完成后更新状态为 `done`
- 执行失败后更新状态为 `error`
- 暂停成功后更新状态为 `paused`

### MySQL 持久化方向

任务表至少需要覆盖这些数据：

- 任务 ID
- prompt
- storyboard_json
- backend
- provider
- status
- stage
- status_text
- effective_backend
- effective_provider
- error
- bgm_path
- output_path
- cover_path
- template_id
- template_name
- platform
- created_at
- updated_at
- started_at
- finished_at
- deleted_at

原则：

- 二进制文件不进 MySQL
- 图片、BGM、结果视频、中间产物仍落磁盘或对象存储
- 数据库只记录路径和任务元信息

## 任务控制接口规划

### 目标接口

后续需要补两个接口：

- `POST /api/jobs/{job_id}/pause`
- `DELETE /api/jobs/{job_id}`

### pause 语义

- `queued` 任务：直接改为 `paused`
- `running` 任务：标记暂停请求，Worker 在安全点停止，并更新为 `paused`
- `done / error / paused`：幂等返回当前状态

### delete 语义

- `queued` 任务：删除数据库记录和关联文件
- `running` 任务：标记删除请求，Worker 尽快停止，然后删除记录和文件
- `paused / done / error`：删除数据库记录和关联文件

### 当前讨论结论

- 最终需要同时保留两个动作：`暂停` 和 `删除`
- 删除偏向硬删除
- 但运行中任务不应直接粗暴中断进程，应由 Worker 在可控安全点停止后再清理

## 前端 UI 规划

本轮文档中的前端范围只覆盖任务流相关页面和交互，不扩展到整套工作台重构。

### 1. 工作台

- 工作台只保留“生成视频”提交入口
- 不出现“创建新任务”按钮
- 用户点击“生成视频”后，前端只创建真实任务记录
- 创建成功后进入状态轮询，不创建任何空任务或草稿任务

### 2. 任务列表页

- 任务列表页只展示真实生成任务
- 不出现“创建新任务”按钮
- 每个任务项至少展示：
  - 任务 ID
  - 状态
  - stage / status_text
  - provider / 模板名
  - 创建时间
- 每个任务项支持：
  - 进入详情
  - 暂停
  - 删除

### 3. 任务详情页

- 任务详情页负责：
  - 预览视频
  - 查看状态
  - 下载视频
  - 下载封面
  - 下载导出包
  - 暂停
  - 删除
- `done` 状态才显示视频播放器
- `queued / running / paused / error` 只显示状态信息与控制动作

## ffmpeg 链路重构方向

### 本轮结论

- 本轮不移除 ffmpeg
- 本轮也不更新相关代码
- 当前只把 ffmpeg 方向写进文档，后续再讨论实施方案

### 后续目标方向

不再鼓励 ffmpeg 继续承担长期主生成链路。

短期方向是：

- 保留 ffmpeg
- 但逐步把它从“本地图片轮播渲染器”收缩为“后处理与兜底模块”

### 候选职责

后续 ffmpeg 更适合承担这些职责：

- 字幕烧录
- BGM 混音
- 视频封面抽帧
- 导出包处理
- 云端结果的格式转换与后处理

### 需要讨论的核心问题

- 是否彻底移除 `local` 主渲染链路
- 若不彻底移除，是否只保留为 fallback
- 如果未来云端 provider 返回的是整片视频，ffmpeg 是否只负责后处理
- 如果未来云端 provider 返回的是分镜片段，ffmpeg 是否负责最终拼接

## 讨论项

以下问题需要在后续讨论中明确：

- MySQL 表结构最终版
- Worker 抢占任务的锁机制
- 暂停语义的安全点定义
- 删除运行中任务时的清理边界
- 云端 provider 返回的是整片还是分镜片段
- ffmpeg 是否保留部分 fallback 渲染能力
- `auto` 模式是否继续保留，还是收缩为显式 `cloud/local`

## 下一阶段实现路线图

### 第一阶段

- 去掉任务页“创建新任务”入口
- 任务只由“生成视频”创建
- 统一页面文案与状态语义

### 第二阶段

- 引入 MySQL
- 用数据库替换 `JobStore`
- 引入独立 Worker 轮询数据库中的待执行任务

### 第三阶段

- 增加暂停/删除接口
- 任务列表页和任务详情页接入控制动作

### 第四阶段

- 梳理 `cloud / local / ffmpeg` 的职责边界
- 决定 ffmpeg 最终只做后处理，还是保留有限 fallback 渲染能力

- `done` 任务能正常预览视频
- ffmpeg 重构后，本地模块至少仍能完成后处理职责，不影响导出链路
